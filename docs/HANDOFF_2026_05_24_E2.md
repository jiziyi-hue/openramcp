## 4. 重要 prior art: OpenRA-RL

仓库: https://github.com/yxc20089/OpenRA-RL — 玩家说"看过, 效果一般".

### 4.1 架构差异 (深度对比)

| 项 | OpenRA-RL | 本项目 (openra_mcp) |
|---|---|---|
| 通信 | gRPC + FastAPI + protobuf, headless Docker | TCP newline-JSON, 真 GUI 玩家 |
| 命令粒度 | per-unit + group + structure | high-level intent (15 个) + squad |
| MCP 工具数 | **48 atomic** | 30 (15 战略 intent + 情报 / 编组 / 生命周期) |
| LLM 算坐标? | **是** (`target_x:int, target_y:int`) | **否** (interpreter 翻 named target) |
| LLM 选 ids? | **是** (`unit_ids: str` "all_combat" / "type:e1") | **否** (filter / group / named force) |
| Approach (flank / charge) | 无 | 有 (interpreter 算 waypoint) |
| Alert state / Objective | 无 | 5 alert state + 5 objective |
| 经济 | LLM 也管 (`produce`, `place_building`, `sell`) | LLM **零经济** (玩家 UI 管) |
| 玩家 in-loop | 否, 纯 RL agent benchmark | 是, 战术参谋帮玩家 |
| Daemon 跑循环 | 无 (LLM 每 tick 重发) | 有 (mission 注册一次 daemon 跑) |
| paper | **无** (只有 leaderboard) | 计划写 |

### 4.2 借鉴点 (可低成本采纳)

| OpenRA-RL 特性 | 借鉴价值 | 怎么用 |
|---|---|---|
| `unit_ids: "all_combat"` selector 字符串 | ⭐⭐ | 给 `force.kind` 加 string shortcut 便利层 |
| `command_group(name, command)` 一句话指挥组 | ⭐⭐⭐ | 你有 `assign_to_group` 但**没** command_group — 加 1 个轻量工具 |
| `batch(actions: list[dict])` 一 tick 多操作 | ⭐⭐⭐⭐ | **强烈推荐**: 加 `batch_dispatch_intent([...])` MCP 工具, 一次 round-trip 发多个 intent. 钳形/佯攻偷家可一次发 |
| `plan(steps)` 步骤序列 + 状态刷新 | ⭐⭐ | 你 Phase B5 watcher 概念类似 — 可考虑 LLM 输出 "wait 5s then X" 序列 |

### 4.3 **不**借鉴的部分

- 48 atomic — 你**故意更抽象**, 别学
- LLM 算坐标 — 违反"LLM 不做数值"铁律
- LLM 管经济 — 违反"玩家拥经济"铁律
- per-unit 命令 — 违反"daemon 拥循环"铁律

### 4.4 paper 卖点 (相比 OpenRA-RL)

OpenRA-RL **无 paper**. 3 axes 你独占:
1. ✅ 真 RTS engine + C# trait injection (它也有, 但你是真玩家窗口)
2. ✅ Human-in-loop chief-of-staff (它是 RL benchmark, 没人)
3. ✅ 高抽象 intent DSL (它是 48 atomic, 你是 15 intent + interpreter offload)
4. ⚠️ MCP 协议 — **两边都用**, 不再独占 (减分项, 但你 MCP 设计更高层)
